# 外部音源（网易云 / QQ 音乐）歌单导入与播放 —— 调研与方案留档

> 状态：**调研完成，未开工**。已定播放路线（浏览器直连直链）+ 已定下一步（先验证直链可播性）。
> 验证结果出来前不写 fbz 代码——验证可能直接推翻「浏览器直连」前提。

## 1. 需求

- 把用户的 QQ 音乐 / 网易云歌单导入 fbz，能播放。
- 作为**插件**接入；**用户用自己的账号 / 自己的 VIP** 登录。

## 2. 关键结论（先读这条）

- **Mineradio（github.com/XxHuberrr/Mineradio）不直接用**：GPL-3.0（会传染 fbz 开源）+ Electron 桌面单机形态（登录态存本机，与 fbz 多用户服务器模型不符）。可参考其思路，不搬代码。
- **合规**：用自己账号 + 自己 VIP 访问自己有权访问的内容，不涉及绕过付费墙。风险由部署者自担，本方案不做凭据抓取 / 付费墙绕过。
- **事实校正**：QQ / 网易云**没有面向第三方播放的官方开放 API**。常用的（如 `NeteaseCloudMusicApi`）是**逆向的非官方接口**（曾被 DMCA 归档）。影响不在合法性，而在**工程稳定性**：接口随时被改 / 封、需定期刷 cookie、有频率限制与 IP 封禁风险。
- **它不是「一个插件」能装下的**：插件层只能当「音源适配器」，但要跑起来必须先在 fbz 核心补三块能力（见 §5）。

## 3. fbz 现有插件系统的限制（为什么纯插件做不到）

依据 `docs/plugin-system.md`：

- 插件拿不到原始 DB，只能调受控 Host API。现有 Host API 仅：KV 存储、`library.read`（只读媒体）、`metadata.write`、`notification.send`、marker/artwork。**没有「导入外部曲目为可播放媒体」的写口**。
- 插件不能旁路媒体库权限。
- 出站 HTTP 只有 `http` runtime 插件能做，且必须命中 `PLUGIN_HTTP_ALLOWED_HOSTS` 白名单——不能让插件自由连第三方接口。
- 无音频代理 / 取流能力：fbz 播放 = 本地文件 + ffmpeg 转码 + HLS。外部源取流是另一条摄取+播放链路，宿主里不存在。
- 插件 config 是**管理员级**的，不是每用户的——没有 per-user 第三方账号 / 凭据存储。

**插件能扮演的角色**：`http` runtime 的「音源适配器」（用户自己跑的小服务，包住那个非官方 API），fbz 受控调用它做：歌单 → 曲目清单、曲目 → 直链解析。凭据库 + 外部音源 + 播放路径仍在 fbz 核心侧。

## 4. 已定路线

- **播放**：浏览器直连直链（后端用账号解析出直链，前端 `<audio>` 直接播；服务器零流量）。
- **下一步**：先验证直链可播性，再动 fbz 代码。

## 5. 若验证通过，核心侧要建的三块（都不是插件层能补的）

1. **per-user 加密凭据库**：仿 `notification_target_secrets` 的加密模式，但做成 per-user 的外部账号凭据存储（「用户自己登录」的落点）。
2. **外部音源抽象（external track）**：`media_items` 是「本地文件 + ffmpeg」，网易云曲目无本地文件、是「外部曲目 + 时效直链」，需新的一等概念，不塞进 media_items。
3. **播放路径**：浏览器直连方案下，前端播放态要能接受外部直链 URL（现有底部播放栏 + 原生 audio 直出流已具雏形，见 [[music-playback-audiobar]]）。

## 6. 待验证：直链可播性（阻塞项，只能用户自己跑）

只能用户用自己账号 + 自己起的非官方 API + 自己浏览器/网络环境跑。四个独立失败模式，任一中招就退回「服务器代理转发」：

1. **IP 绑定**：后端解析出的直链在用户浏览器（可能不同公网 IP）能否播。网易云 CDN 通常不锁 IP、只限时；QQ 更严（vkey/guid）。
2. **混合内容（最可能翻车）**：网易云 CDN 直链常是 `http://`。若 fbz 前端是 `https://`，浏览器直接拦截 http 音频。**纯协议问题，与 IP 无关。若 fbz 生产是 https，此路基本当场出局。**
3. **Referer/Origin 校验**：部分 CDN 看 Referer，浏览器带 fbz origin 可能被拒。
4. **Range 请求**：能播不代表能 seek（拖动进度）。

**注意**：`<audio>` 播跨域音频不需要 CORS（不加 crossorigin 时），别被 `fetch()` 的 CORS 错误误导，以 `<audio>` 实际出声为准。

### 诊断工具（不进仓库，一次性）

存成 `stream-test.html` 本地打开，贴入后端解析出的直链，逐项判读上面四坑。工具全文见对话记录（含 canplay/playing/error 事件判读、混合内容检测、拖动测试）。判读表：

| 现象 | 结论 |
|---|---|
| `playing` 出声 + 能拖动 | 浏览器直连可行，按此设计 |
| 混合内容警告（https 页 + http 直链） | 生产(https)下不可行，退回服务器代理 |
| error code 2/4（NETWORK/SRC） | IP 锁 / Referer 拒 / 直链过期 → 退回服务器代理 |
| 能播但拖不动 | Range 不支持，代理时要补 |

**直链来源**：本地起维护中的 `NeteaseCloudMusicApi` fork，登录账号后调 `/song/url/v1?id=&level=` 取 `data[0].url`。

## 7. 恢复推进时的第一件事

问用户两个问题定架构：
1. 直链可播性验证结果（哪几坑中了）——决定「浏览器直连」还是「服务器代理」。
2. fbz 生产是 http 还是 https——若 https，混合内容基本淘汰「直连 http 直链」，省掉一半验证。

## 8. 稳妥的降级方案（若直链路线不可行）

只导入歌单的**曲目清单**（歌名/歌手），后端按名 + 拼音匹配本地 `media_items`，命中的生成 fbz 本地歌单。播放始终是本地文件，零 ToS/取流/直链风险，且天然复用已完成的拼音搜索（见 [[pinyin-search-keys]]）。这个可作为 `library.read` + 一个新「按名批量匹配」Host API 的插件，符合现有沙箱模型。
